## Summary

Add short numeric IDs for FE-visible records while keeping UUIDs as the internal primary keys.

## Scope

- Add database migration support for `display_id BIGINT` on existing FE-visible records.
- Backfill existing rows in a stable order and keep future values generated by table-specific sequences.
- Update Go models/stores so repositories can return `DisplayID` with the internal UUID.
- Update development docs so future service, provider, order, invoice, transaction, and log tables follow the same pattern.
- Out of scope: replacing UUID primary keys, changing auth rules, or building FE screens.

## Acceptance Criteria

- UUID remains the primary key and FK target.
- Existing FE-visible tables have `display_id BIGINT NOT NULL UNIQUE`.
- Backfilled values start at 10000 per table.
- Stores that scan FE-visible records include `DisplayID`.
- Docs explain when to use UUID versus numeric display IDs.
- Validation passes: `make fmt`, `make test`, `make build`, `make migrate-validate`.

## Notes

- `display_id` is for UI/support/search only.
- Do not authorize or join by `display_id` alone; resolve under tenant/context scope first.
- Pure junction/system tables do not need display IDs unless they become first-class UI records.
